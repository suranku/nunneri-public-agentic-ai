The Problem Before

Provider lock-in

Assets often work only for one assistant and cannot move cleanly across teams.

Unclear ownership

Feedback arrives without enough context to triage, prioritize, or release.

Manual drift

Repeated copies diverge when commands, agents, and docs are maintained separately.

The Solution

Canonical assets live once under assets, then adapters generate Claude, Codex, Gemini, and open-source outputs.

Interactive

1. Intake

Capture the ticket, environment, symptom, expected behavior, actual behavior, user impact, and known missing information.

2. Context Load

Read provider-neutral context first, then provider context such as CLAUDE.md, AGENTS.md, or GEMINI.md, followed by project build and test docs.

3. Classification

Classify the problem as code, config, data, schema, infrastructure, dependency, framework, documentation, asset metadata, or unknown.

4. Evidence Collection

Gather stack traces, logs, reproduction steps, code search results, git history, configuration differences, and adapter output as needed.

5. Root Cause Analysis

State the root cause as a testable claim and separate confirmed evidence from assumptions, risks, and rollback notes.

6. Gate 1: RCA and Fix Plan Approval

Stop before edits. Present the root cause, evidence, fix plan, test plan, expected files, and risks, then wait for explicit approval.

7. Test-First Fix

After approval, create a failing test or equivalent validation, then implement the smallest correct fix.

8. Validation

Run stack checks and asset checks such as validate, adapter build, portal manifest build, and reference doc sync where relevant.

9. Gate 2: Summary and Release Impact

Stop before commit, push, PR, tag, or publish. Present validation results, provider impact, release impact, compatibility notes, and recommended commit or PR text.
